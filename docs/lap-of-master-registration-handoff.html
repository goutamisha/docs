Lap of Master 2026 — Marketing → Registration handoff

Question: The marketing site (iso_v2) is static and has no form. The registration/booking form lives in a separate 3-repo system (prs-frontend + prs-facade + prs-backend). For the "Lap of Master" event (and future events), how should the Register Now CTA hand the user off to the form? Two approaches were proposed:

  • (A) Keep everything static in iso, redirect to the form URL with query params.
  • (B) Expose an API from prs-facade, let iso POST step 1 directly, then redirect to the form URL with a step + booking token.

Answer (short): The PRS form system is being built new — so design the handoff as a clean two-endpoint contract, not around the legacy SPA. Tier it by event type:

  • Standard events (no bespoke design): (A) redirect to /event/<slug>-<flowId> (cheapest; PRS owns the UI).
  • Bespoke-designed events like Lap of Master (custom seat map): (B) contract-driven — the form team exposes an inventory JSON API, ISO renders its own Figma UI from it, and on Continue ISO submits to get booking_token + step_token and redirects to PRS. This is not reimplementation — PRS owns booking state/tokens/payment, ISO owns presentation.

Avoid the widget/iframe embed (C) for this design: the Figma seat-map is ISO's bespoke UI, so the widget would force the form team to rebuild ISO's design in their repo, and it requires fragile postMessage glue (open/close/resize/ complete) across two repos.

Note on legacy facts: earlier analysis of the as-built prs-frontend/prs-facade (static x-system-token in the client bundle, booking created implicitly on the generic step POST, widget on MemoryRouter) describes the old system. The new system is unbuilt, so those are not constraints — they're anti-patterns to design away from.

How the PRS form system actually works

From reading prs-frontend, prs-facade, prs-backend:

prs-frontend — React 18 + Vite SPA, self-driving multi-step form.

  • Public entry route: /event/:eventSlug where eventSlug ends in the numeric flow id: flowId = eventSlug.split('-').pop() (src/helper/extractPageNameAndEventId.ts). So a link is /event/lap-of-master-<flowId>.
  • Reads query params on entry: utm_source/medium/campaign/term/content, lang, prefill_token, jwt_token_iprs (SSO), and resume params token, step_token (src/hooks/useFormDetails/useFormDetailsUrl.tsx).
  • Resume/deep-link routes: /event/:eventSlug/:bookingToken/register, /event/:eventSlug/:bookingToken/:stepToken.
  • Ships a widget build: prs-form.bundle.js, mounted via <div id="prs-form" data-props='{"slug":"…","eventLanguage":"…","theme":"…"}'> (src/mfa/index.tsx).

prs-facade — Python Litestar BFF.

  • A booking is created implicitly on the first POST /flows/{flow_id}/steps/{step_key}/form (body FormStepSubmissionData { form_data, booking_id? }); the response returns data.booking_data.booking_token (a UUID). There is no POST /registrations create endpoint.
  • Every booking/flow/step/form endpoint requires the header x-system-token == PRS_SYSTEM_STATIC_TOKEN (Utils/middleware.py, exclude list in Utils/configs.py). CORS is allow_origins=["*"], so CORS is not the blocker — the static token is. (Note: this same token is already baked into the public SPA/widget bundle as VITE_APP_STATIC_TOKEN, so it is a coarse gate, not a true secret.)
  • The facade itself builds the canonical handoff URL {FRONTEND_DOMAIN}/event/{flow_id}/{booking_token}.

prs-backend — Odoo. Owns the DB/business logic/payment; facade reads/writes the Odoo Postgres DB directly via asyncpg.

Takeaway: the whole system is designed for the frontend (form engine) to render the dynamic steps and create the booking. Step data (categories, prices, seat availability) is server-driven and changes per event.

iso_v2 already has the pieces

  • CTA + external link is CMS-driven. GpEventBanner / GpRegisterCta render a Chakra <Link href={cta.href} isExternal={cta.openInNewTab}>; the href comes from the Dato simple_button.linkUrl field (services/dato/mappers/gpEventBanner.ts → RawButton { buttonText, linkUrl, openInNewTab }). No code change is needed to point a CTA at an external registration URL — it's content.
  • The PRS widget is already embedded today. sections/registerTr/RegisterForm.js loads config.INLINE_FORM_G_EU_PRS (…/widget/prs-form.bundle.js) and renders <Box id="prs-form" data-props='{"slug": "<flow>", "eventLanguage": "en_US", "theme": "isha-foundation"}' />, reading the flow from ?flow=. Option C is a proven, existing pattern in this repo, not new infrastructure.
  • Drawer/overlay primitives exist. Chakra Modal/Drawer + the RegisterPopupVisibilityContext (useAnimatedDisclosure) pattern can host the side-sheet.
  • Routes are /{region}/{language}/<program> with regions in | us | global.

The contract (recommended) — inventory API + ISO UI + submit→tokens→redirect

Because the system is new and the seat-map UI is ISO's bespoke design, the cleanest generic handoff is a two-endpoint contract between the teams. PRS owns data + booking state; ISO owns presentation.

1. Inventory (read) — public, cacheable, CORS-enabled. This JSON is "what you see in the Figma".

GET {PRS_API}/v1/flows/{flowId}/inventory?lang=&region=&currency=
→ {
    event:   { flowId, slug: "lap-of-master", title, currency },
    steps:   { current: 1, total: 2 },
    seatMap: { zones: [ … ] } | null,
    categories: [
      { id, name: "Yogeshwara", price: 50000, currency: "INR",
        stay: "Executive AC twin sharing", hall: "Seating in Adiyogi Alayam Hall",
        distance: "5-min walk to program area",
        availability: "available" | "limited" | "soldout", seatsLeft }
    ]
  }

2. Submit step 1 (write) — authoritative, returns tokens + the next URL.

POST {PRS_API}/v1/flows/{flowId}/steps/{stepKey}/submit
body: { form_data: { categoryId, seatId? }, lang, region, utm }
→ { bookingToken, stepToken, nextUrl }      // nextUrl is built by PRS, not ISO

On Continue, ISO POSTs the selection, receives bookingToken + stepToken, then window.location = nextUrl. PRS takes over for step 2 (personal details + payment).

Three rules that make this safe and generic:

  1. No long-lived secret in the browser. Route the POST through a thin Next.js route handler in ISO (/api/register/[program]) that holds any credential server-side and forwards to PRS. The inventory GET stays public.
  2. The submit is authoritative. ISO's loaded inventory can go stale (a bay sells out before Continue), so the submit re-validates availability/price/eligibility and returns a typed error (SOLDOUT, PRICE_CHANGED) that ISO shows and then refreshes inventory. Correctness on the server, presentation on ISO — standard checkout design.
  3. PRS owns URL format. Returning nextUrl means ISO never constructs PRS URLs; the two systems evolve independently.

The three options, scored

Option A — Static + redirect ✅ recommended default

REGISTER NOW → https://<prs-domain>/event/lap-of-master-<flowId>?utm_*&lang=<lang>.

  • Effort: content-only. Set linkUrl in Dato. Zero code, zero backend changes.
  • Pros: matches the system's intended design; form engine renders the bay-selection step and owns all of step 1→payment→confirmation; UTM/lang/SSO prefill all already supported; nothing to maintain when the step schema changes.
  • Cons: the bay-selection step shows as a full PRS page, not as an overlay on the marketing hero — a visual deviation from the Figma side-sheet.

Option B — Contract-driven: inventory API + ISO UI + submit→tokens→redirect ✅ recommended for bespoke events

ISO renders the Figma seat-map/category UI from the inventory JSON; on Continue it submits the selection (via an ISO Next.js route handler) and gets { bookingToken, stepToken, nextUrl }, then redirects to PRS. See The contract section above for the exact endpoints.

  • Effort: moderate–high, split cleanly across teams: form team builds 2 endpoints; ISO builds the UI it designed.
  • Pros: pixel-perfect to the bespoke Figma (ISO renders its own design); native popup open/close — it's all ISO React, no postMessage; clean team boundary (PRS = data + booking state, ISO = presentation); secure (no secret in the browser; submit is server-authoritative); generic via a versioned contract.
  • Cons: two teams must agree on the inventory + submit schema; ISO must handle server-returned errors (SOLDOUT/PRICE_CHANGED) and refresh inventory; ISO owns a (thin) Next.js route handler.
  • Not reimplementation: ISO does not own form logic, validation rules, pricing truth, or booking creation — it renders JSON and posts a selection. The server remains authoritative.

Option C — Embed the PRS widget/iframe in a side-sheet ❌ not recommended for this design

REGISTER NOW → open a Chakra Drawer that mounts the PRS widget/iframe; communicate via window.postMessage.

  • Why it doesn't fit: the seat-map is ISO's bespoke design, so the form team would have to rebuild ISO's UI inside the widget (wrong ownership, duplicated work, styling drift). And open/close/resize/complete must be wired over fragile cross-repo postMessage glue, plus iframe height/SSO/CSS/mobile concerns.
  • When C would make sense: a standard event whose form has no bespoke design and PRS's generic look is acceptable — then the widget avoids ISO building any UI at all.

Recommendation

  1. Standard events → Option A (redirect). Store the flow id in the Dato program/banner entry and build the CTA linkUrl as https://<prs-domain>/event/<slug>-<flowId> with utm_* + lang passthrough. Lowest-risk; PRS owns the whole UI. Zero backend work.

  2. Lap of Master (and other bespoke-designed events) → Option B (contract-driven). Form team exposes GET .../inventory and POST .../submit (returning bookingToken + stepToken + nextUrl); ISO renders the Figma UI from the inventory JSON and, on Continue, submits via a Next.js route handler and redirects to nextUrl. This honors the design exactly, keeps popup open/close native, and keeps booking authority on PRS.

  3. Avoid Option C (widget/iframe) for this design — the bespoke UI belongs in ISO, and postMessage glue is avoidable coupling. Reserve it for non-bespoke events only.

Generalizing to "upcoming event/program"

The handoff is the same two contracts for every event; only the rendering tier differs:

  • Per-event config = the PRS flowId (+ slug stem + which tier), stored as CMS fields on each program entry in Dato.
  • Tier A (redirect): build the URL from one helper, buildPrsEventUrl({ slugStem, flowId, region, language, utm }) → /event/<slugStem>-<flowId>?lang=<language>&utm_*.
  • Tier B (contract): one reusable ISO flow — fetch GET .../inventory, render the (event-specific) selection UI, POST via the shared /api/register/[program] route handler, redirect to the returned nextUrl. The selection UI component varies per bespoke design; the data-fetch + submit + redirect plumbing is shared.

Invariant across both tiers: PRS owns booking state, tokens, payment, and the canonical URL; ISO owns presentation when there's a custom design. A new event ships by editing CMS content (+ a UI component only when it has a bespoke design).

Scalability verification (query-param redirect vs JSON API)

Verified against the real prs-frontend / prs-facade / prs-backend source.

Passing inventory selection back via query params — not viable, doesn't scale.

  • PRS has zero raw-query-param → form-value plumbing. No ?category= / ?seat= / ?inventory_id= reader exists. All pre-selection is token-based (prefill_token, step_token+bookingToken, SSO prefill_reference), and every token requires a server-side artifact that already exists — there is no endpoint to mint a prefill token from external data (Helpers/api/step/get_form_step_details_api_helper.py:182).
  • Inventory is chosen by a server-side Jinja selection_expression engine keyed on the participant's own answers, not on URL params (Helpers/form_helper.py:206).
  • Unknown query params are silently ignored (Litestar binds only declared params; no whitelist), and one-shot params (utm_*, jwt_token_iprs, join_booking) are stripped from the URL on first render.
  • So a redirect-with-selection needs new PRS code regardless (a param reader seeding the inventory_id field's default_value; the server half is coincidentally present — populate_inventory branches on inventory_id != "" at get_form_step_details_api_helper.py:155, and verify_inventory_belongs_to_event exists). Every future field ISO wants to pass = another bilateral param + reader. Stringly-typed, unvalidated, URL-length-limited, undocumented.
  • Deeper flaw: query params only carry the selection back to PRS; they don't let a static ISO page display live inventory — that data would have to be hardcoded/CMS'd and go stale against PRS's live availability/pricing.

A JSON API — extends with little/no code, this is the scalable path.

  • Facade read responses are untyped Dict[str, Any] sourced from JSONB blobs (form_json, inventory_json) and raw asyncpg dicts. msgspec.Struct types exist only for request bodies (Utils/structs.py) — never for reads. So exposing more data (program description, eligibility, add-ons, discounts) is usually config in Odoo, not facade code; a new DB column is a one-line SELECT.
  • Submit already returns booking_token + next_step (Helpers/api/step/submit_form_step_api_halper.py:303) — clean handoff seam.
  • Two one-time costs: (1) the form GET is a render artifact, not a clean catalog (prices/seats merged into form-field default_values), so add a purpose-built read endpoint rather than scraping the form schema; (2) no public auth model — everything is behind one static x-system-token (CORS is * but moot), so add a public/read-only endpoint or proxy via ISO's Next.js route handler.

Verdict: query-param redirect is acceptable only for the trivial "ISO shows nothing, bounce to /event/<flowId>" case (Tier A). The moment ISO must display inventory or grow the page — the stated future direction — query params break down and the JSON API is required. Make the API a page-data contract, not just inventory, so it absorbs future fields without a new round-trip of bilateral changes:

GET {PRS_API}/public/v1/flows/{flowId}/page-data?lang=&region=&currency=
→ { event, steps, categories[], seatMap, /* extensible: descriptions, eligibility, addOns, discounts, … */ }
POST {PRS_API}/public/v1/flows/{flowId}/steps/{stepKey}/submit → { bookingToken, stepToken, nextUrl }
