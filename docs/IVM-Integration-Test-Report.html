In one paragraph

Three operational areas of the IVM dashboard were wired to the existing IVM API and tested end to end. Testing runs at two levels. The first is an offline contract layer (Vitest) that proves every screen builds the correct request and reads the correct response, with the network mocked. The second is a live end-to-end run that drives the actual running stack the way a browser does, against a real PostgreSQL database seeded with demo data. Both are green: 434 of 434 Vitest checks pass and 45 of 45 live checks pass, including reads, writes, the CSRF guard and every page route. The data used is real database content created by the repository's seed commands. It is demo data, not production data. The three scheduling screens that began on placeholder data (My Bookings, the Reschedule-needed lane, the Over-cap lane) are now bound to live endpoints, and the work is merged to the uat branch and deployed to UAT.

2 Testing approach, two levels

Level 1, offline contract (Vitest)
Network mocked. Proves each screen builds the right URL, HTTP method, query parameters and request body, and maps the response correctly. Runs in CI with no backend. This is the "is the wiring correct" layer.
Level 2, live end to end
A repeatable script drives the real running system the way the browser does, against a real database. Exercises reads, writes, the security guard and every page route, then restores a clean data state. This is the "does it actually work" layer.
Honest distinctionLevel 1 uses no data at all (the network is mocked). Level 2 uses a real running stack and a real PostgreSQL database, populated with demo data from the seed commands. It is not production or customer data, and no production data was touched.

3 The real-data test, what it ran against

The live run follows the exact path a browser takes. Same-origin calls hit the Next.js BFF proxy, which reads the session cookie and injects the bearer token, then forwards to the Django API, which reads and writes a real Postgres database.

browser / curl
  -> http://localhost:9100/api/v1/*        Next.js frontend (IVM-Backend)
  -> app/api/v1/[...path]/route.ts          BFF proxy: reads ivm_session cookie, injects Bearer JWT,
                                            requires the x-ivm-auth CSRF header on mutations
  -> http://localhost:8000/api/v1/*         Django API (ivm-api), JWT auth + role and scope guard
  -> PostgreSQL  db=ivm @ 127.0.0.1         real rows

6 Real defects found and fixed during testing

The live run earned its keep. Six issues only a real end-to-end pass would surface, the last two during the round that rebound the scheduling screens.

IssueCauseFix
Attendance list calls failed through the proxyThe frontend uses a BFF proxy, not a rewrite. It strips a trailing slash, and the API serves its collection routes with one. An earlier trailing-slash workaround was wrong for this path.Send the no-slash form. The BFF follows the API redirect for reads server-side.
Re-seed could not restore stateThe shared reset deletes participants, but the attendance tables hold protected references to them, and the swipe and audit tables are append-only (deletes are blocked by a trigger).Re-seed now truncates the attendance and audit tables first (truncate bypasses the trigger and the reference ordering), then resets.
Auth header ignoredThe proxy reads the session cookie and injects the token itself. A bearer header passed by hand is dropped.Test through the session cookie, the way the browser authenticates.
Propose returned a conflictNot a defect. The "one volunteer per VRF" business guard fired on a volunteer already linked.The test now treats the guard as a valid outcome and confirms a clean propose on fresh data.
New scheduling endpoints returned a server errorThe four new fields (caller-unavailable, interviewer gender / overseas / timezone, caller call-window) had unapplied database migrations on the running environment.Applied the pending migrations. The endpoints then returned data.
Re-seed broke after the API updateWhen the API was updated to add the new endpoints, the local seed helper that tops up the demo data was removed from the working tree.Recreated the helper. Re-seed restores a clean state again.

8 How to reproduce

# 1. Seed the real database (idempotent, safe to re-run)
cd ivm-api
.venv/bin/python manage.py seed_demo
.venv/bin/python manage.py seed_attendance_demo
.venv/bin/python manage.py seed_attendance_extras

# 2. Run the stack: API on :8000 (DEBUG on), frontend on :9100

# 3. Live end-to-end run (real data, mutates demo rows, then re-seeds)
cd ../IVM-Backend
bash scripts/ops-integration-e2e.sh

# Offline contract tests (no backend needed)
pnpm exec vitest run
To click through it yourselfOpen http://localhost:9100/auth/dev?as=admin@ivm.dev to sign in as the persona that can see and act on every Seva and Attendance screen, then use the sidebar. For My Availability use interviewer@ivm.dev, who owns the published slots.

9 Caveats and known limitations

  • Demo data, not production. Everything was tested against seeded demo data. No production or customer data was read or changed.
  • One screen still on placeholder data. The Seva dashboard recent-activity feed has no API endpoint yet, so it reads a JSON file. Everything else is bound to live endpoints.
  • The seed helper lives in the API repo and is untracked. seed_attendance_extras.py is a local helper in ivm-api. It is not part of the frontend push and is removed whenever the API code updates, so the backend team should commit it if it is to persist.
  • One write is not yet bound through the proxy. Creating a single one-off session posts to a collection route that the proxy cannot reach cleanly (the API will not append a slash on a POST). Every other action and all reads work. The fix belongs on the API side (a no-slash alias) or by using the recurring-series endpoint instead.
  • Names show as identifiers in two places. Attendance records and excusals display Participant #id because those API responses carry only the id, not a name. This is an API response shape, not a test gap.
  • Repository baseline. The frontend repository had pre-existing type and lint notes in test and generated files before this work. The new files add none.
