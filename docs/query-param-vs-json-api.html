How should the marketing site hand users off to the registration form?

Two ways to connect the marketing page (ISO) to the new registration/booking system (PRS): a simple link with query parameters, or a data API the marketing site reads and submits to. This note compares both — for both the technical and business teams.

Event: Lap of Master 2026 Decision note Verified against PRS source code

In one line

The query-param link is fine only when the marketing page shows nothing and just sends the user straight to the form. The moment the marketing page needs to show real options (rooms, prices, seats) or grow over time, the JSON API is the scalable choice — and it's the recommended direction.

What each approach means (plain language)

Think of booking the Lap of Master program like booking a hotel room from a travel site.

Approach A

Query-Param Redirect

Marketing page is static → user clicks Register → we send them to the form with details tacked onto the web link (e.g. ?category=…).

Like a travel site that doesn't show room options itself — it just bounces you to the hotel's own booking page. Anything we want to "pass along" has to be squeezed into the web address.

Pros
  • Simplest to launch for the basic "just send them to the form" case
  • No data work on the marketing side
Cons
  • The marketing page can't show live rooms/prices — that data isn't available to it
  • Any info we hardcode goes stale (price/availability changes)
  • Every new detail to pass = new code on both teams, every time
  • Web-address contract is fragile (length limits, no validation)
Doesn't scale as the page grows
Approach B

JSON Data API

Form team exposes a data feed → marketing page reads it, shows the options in our own design → on Continue it submits and forwards the user to the form.

Like a travel site that shows live rooms and prices in its own layout, lets you pick, then hands you to the hotel to finish payment. The marketing page stays in control of the look and always shows current data.

Pros
  • Shows live, accurate rooms / prices / availability
  • Marketing page renders the exact Figma design
  • Adding more info later is usually config, not new code
  • Clean ownership: form team owns data & booking, marketing owns the look
Cons
  • Two teams agree on the data shape once
One-time setup cost (not a recurring drawback) Form team builds a public data + submit endpoint — paid once, then reused for every future event. Note: this is not unique to this approach — the query-param redirect also needs new PRS work (PRS has no way to read these params today), so it doesn't tilt the comparison. The recurring costs all sit on the query-param side.
Scales cleanly · recommended

Verdict & recommendation

Use the JSON Data API as the scalable path. Keep the query-param link only for the trivial "show nothing, send to the form" case.

  • Standard events (no custom design): a simple redirect to /event/<slug>-<flowId> is enough.
  • Designed events like Lap of Master (custom seat map): the marketing page reads a data feed, renders the design, submits, and forwards the user with their booking token.

Ask the form team for one reusable, public, versioned “page-data” feed (not just inventory — so it absorbs future fields) plus an authoritative submit that returns the booking handoff:

GET  {PRS_API}/public/v1/flows/{flowId}/page-data?lang=&region=&currency=
  →  { event, steps, categories[], seatMap, /* + descriptions, eligibility, add-ons, discounts … */ }

POST {PRS_API}/public/v1/flows/{flowId}/steps/{stepKey}/submit
  →  { bookingToken, stepToken, nextUrl }   // PRS builds nextUrl; ISO just redirects

Effect on page load & performance

Key point: registration is a popup (side-sheet). The data API runs only when the user opens it — never at page load. So the marketing page loads exactly as fast as it does today.

Initial page load is untouched

The marketing page is static / CMS content. No registration API runs while the page loads, so LCP, speed, and SEO are unaffected. The API calls are deferred to user interaction.

What PRS caches today (verified in code)

  • Already cached (Redis, read-through): flow definitions (event_flow_{flowId}) and event details (event_json_{eventId}_{lang})  (step_helper.py, event_helper.py) — the slow-changing parts of the feed are fast.
  • Deliberately live (uncached): seat / inventory availability is read fresh from the database every request — so prices and seats shown are always current (correct for booking).
  • Not present yet: no HTTP Cache-Control headers and no CDN/edge cache in front of the API. Adding those for the new public feed is an easy, optional speed win for repeat opens.

Why query-params don't scale (technical, verified in PRS code)

PRS has no raw-query-param → form-value plumbing

  • No ?category= / ?seat= / ?inventory_id= reader exists. All pre-selection is token-based (prefill_token, step_token+bookingToken, SSO prefill_reference).
  • Every token needs a server-side record that already exists; there is no endpoint to mint a token from external data  (get_form_step_details_api_helper.py:182).
  • Inventory is chosen by a server-side rules engine on the participant's own answers, not on URL values  (form_helper.py:206).
  • Unknown params are silently ignored; one-shot params (utm_*, jwt_token_iprs) are stripped on first load.

⇒ A redirect-with-selection needs new PRS work regardless, and each future field is another bilateral change. Plus query-params can't let a static page display live inventory at all.

Why the JSON API scales (technical, verified in PRS code)

The PRS data layer is already built to expose more over time

  • Read responses are flexible JSON from JSONB blobs (form_json, inventory_json) and raw DB rows — typed structs exist only for request bodies  (Utils/structs.py). New fields are usually Odoo config, not facade code.
  • The submit path already returns booking_token + next_step  (submit_form_step_api_halper.py:303) — a clean handoff seam.

Two one-time costs: (1) add a purpose-built read endpoint — today's form feed mixes prices into form fields rather than a clean catalog; (2) add a public read auth model — everything currently sits behind one shared static token. Both are one-time and then scale to every event.
