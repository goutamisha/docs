API access: Direct vs BFF

How the browser reaches the IVM API — calling the API directly (the PRS pattern) versus routing through a Backend‑for‑Frontend proxy (the current IVM dashboard pattern). Short comparison, with the decision for IVM's domains.

v1 · ivm-dashboard / ivm-pwa / ivm-api
Verdict for IVM today

Keep the BFF. While the frontends live on *.netlify.app and the API on *.sadhguru.org, the Direct model cannot work in the browser (third‑party cookies are blocked; netlify.app is a public suffix so no shared cookie). The BFF works on these domains and survives the move to real domains.

Later, once the frontends are served under *.ivm.sadhguru.org (same site as the API), Direct becomes a real option — useful if you want cross‑app SSO and the best fit for real‑time + heavy media.

01 The two models

BFF (current IVM)

Browser→ BFF proxy→ API

Browser calls its own origin /api/v1/*. The BFF reads a same‑origin HttpOnly cookie, injects Authorization: Bearer, and forwards. The API only ever sees a Bearer token.

Direct (PRS)

Browser→ API

Browser calls the API directly (different subdomain) with withCredentials. The token rides a parent‑domain HttpOnly cookie (.sadhguru.org) the API reads itself; identity via /auth/me.

03 The domain reality (the deciding factor)

The choice is forced by where the apps are hosted:

  • Frontends: ivm-dashboard.netlify.app, ivm-pwa.netlify.app (UAT: uat--…)
  • API: ivm-api.sadhguru.org (UAT: uat-ivm-api.sadhguru.org)
Why Direct can't work on these domains

1. Different registrable domains. netlify.app ≠ sadhguru.org → a frontend calling the API is cross‑site, so the cookie is third‑party. Safari blocks it; Chrome is phasing it out — even with SameSite=None.

2. netlify.app is on the Public Suffix List. Browsers reject any cookie scoped to .netlify.app, and treat dashboard.netlify.app / pwa.netlify.app as separate sites — so no shared cookie / no cross‑app SSO is possible there at all.

PRS's multi‑domain trick (one facade serving .sadhguru.org and .lingabhairavi.org) works only because, for each brand, the frontend and API share one registrable domain. The same trick can serve .sadhguru.org for IVM — but it can never include netlify.app.

If/when you go Direct, the non‑negotiables

  • CORS: strict origin allow‑list + Allow‑Credentials: true + Vary: Origin. Never * with credentials.
  • CSRF: require a custom header on mutations (forces a preflight your allow‑list denies) or a double‑submit token. The must‑not‑skip item.
  • Cookie: HttpOnly; Secure; SameSite=Lax; Domain=.ivm.sadhguru.org (narrowest parent), reuse PRS's per‑request domain allow‑list.
  • CSP on every frontend (a wider cookie domain makes XSS on any sibling app worse).

Real‑time & heavy media (handle the same way in both models)

  • Don't auth these with cookies. Fetch a short‑lived token (via the BFF or /auth) and pass it to a dedicated WebSocket/SSE service.
  • Media: use presigned URLs straight to storage — bypasses any proxy, works on any domain.
One line

Direct is faster and better for SSO/real‑time, but can't run on the current netlify.app + sadhguru.org split. Stay on the BFF now; revisit Direct/hybrid only after the frontends move under *.ivm.sadhguru.org. Either way, keep the API Bearer‑capable and put expected_version in the body so no CDN can ever strip it.

BFF = Backend for Frontend. Direct = browser → API with a shared parent‑domain cookie (PRS pattern). Generated as an internal architecture note for IVM.
